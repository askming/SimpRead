## 写在前面：一个“不合格”程序员的自白

 
我的技术履历看起来很标准：电子信息工程本科，后来转到计算机科学，再到数据科学、量化管理。从最早学 C 语言开始，到后来工作中大量使用 Python，无论是求学、工作还是创业，我这二十年一直都在跟代码打交道。

 
但我有一个不太“政治正确”的秘密： **我并不擅长也不喜欢写代码。**

 
这不是谦虚，是真实的自我认知。我不是那种能在 IDE 里连续奋战十几个小时、从代码的正反馈中获得快感的性格。很多优秀的工程师能从解决一个 bug、优化一段算法中得到巨大的满足感，但对我来说，写代码更多是实现想法的手段，而非目的本身。这种特质让我在需要深度编码的场景下总觉得力不从心。

  
## 转折点：当 AI 成为你的“高带宽协作者”

 
现在有了 AI 的辅助，我脑子里的那些想法终于可以用一种全新的方式落地 —— **不间断、随时随地、高带宽的协作** 。我不再需要自己敲完每一行代码，而是把精力放在更擅长的事情上：架构设计、需求拆解、方案权衡。

 
这个过程中，我也像很多人一样，尝试了各种工具。有些来自科技媒体的报道，有些来自 Twitter 上 KOL 的推荐。从 GitHub Copilot 到 Cursor，再到 Claude Code，我逐渐找到了最适合自己工作流的组合。

 
最近的一次实践让我真正感受到了这种变革的力量： **我用 Claude Code 加上几个插件，一个人花了大概一两天时间，从前端到后端完成了一个完整的 demo。** 整个过程的完成度和质量，在我的认知里，这原本哪怕在字节，也是需要一个十几人的小团队、投入至少一到两周才能做到的事。

 
这不是夸张，而是真实发生的效率跃迁。而这一切的核心，就是我接下来要分享的这套插件组合。

### 二、这套组合如何完整接管一个项目

 
让我用一个实际的工作流来展示这套工具的协作方式：

 
```
【场景起点】你说:"我想做一个 XX 功能"         │         ▼ 【Superpowers 自动接管】   ├─ 不急着写代码 —— 先通过对话问清楚你真正想要什么   ├─ 把需求文档(spec)拆成可以逐段确认的小块给你看   ├─ 等你签字确认后,生成足够细致的实现计划   └─ 启动多个子代理并行推进每个任务      · 边做边自检,不需要你持续盯着      · 可以自主运行 2 小时不偏离轨道         │         ▼ 【gstack 接管"写完之后"的所有事】   ├─ /review          → 多代理并行检查代码质量   ├─ /ship            → 创建 PR + 合规检查   ├─ /land-and-deploy → 合并代码 + 自动部署   ├─ /canary          → 上线后灰度监控   └─ /cso             → 月度安全全扫描         │         ▼ 【gstack 的设计流水线】(如果需要 UI)   ├─ /design-shotgun → 一次生成 3 种视觉方案   ├─ /design-html    → 选中方案后落地成生产级 HTML/CSS   └─ /design-review  → 视觉质量审查         │         ▼ 【Claude 内置 Memory 贯穿全程】   └─ 你的偏好、决策、项目背景跨会话保留      不需要每次重新解释上下文
```

 
**整个过程中，你的核心动作只有两个：说想做什么 + 确认 spec。**

 
这就是为什么我能一个人在两天内完成原本需要小团队的工作 —— 不是因为我突然变成了超级程序员，而是因为这套工具把我从“写代码”的执行层解放出来，让我专注在“想清楚要做什么”的决策层。

### 四、Claude Mem：为什么我选择不安装？

 
Claude Mem 在社区里也很火，有 7.1 万 star。它做的事情很明确：每次对话结束后，用 Claude API 自动压缩会话内容，提取关键信息，存成结构化文件；下次开新对话时，把相关记忆注入进来，让 AI 记得你上次说过什么。

 
听起来很有用，但我为什么不装？

 
**因为 Claude Code 已经有内置的 auto-memory 系统。** 它在对话中自动识别值得记忆的信息（用户偏好、项目决策、反馈教训）并持久化，跨会话直接生效。工作原理和 Claude Mem 类似，但更深度集成。

 
如果两套记忆系统并存，会带来三个问题：

 
1. 
**记忆分裂** —— 同一件事在两个系统里可能有不同版本，AI 不知道该信哪个


2. 
**注入冲突** —— 两个系统都会在对话开始时注入上下文，可能互相矛盾，也会消耗更多 token


3. 
**维护成本** —— 出现记忆错误时，你不知道是哪个系统的问题



 
**结论：Claude Mem 解决的问题，内置 auto-memory 已经解决了。个人感觉装了只会更乱，所以我卸载了。**

  
### 五、我的实际使用节奏

 
经过这段时间的实践，我总结出了一套比较顺畅的工作流：

 
```
【新功能开发】   → 什么都不用说,Superpowers 自动接管   → 对话确认需求 → 生成计划 → 执行  【UI 需要设计】   → /design-shotgun 出 3 个方案   → 选一个 → /design-html 落地成代码  【代码写完】   → /review 最终审查   → /ship 创建 PR  【部署上线】   → /land-and-deploy 合并并部署   → /canary 监控 24 小时  【月度例行】   → /cso 安全扫描   → /retro 复盘总结  【记忆管理】   → 内置 auto-memory 自动处理   → 不用手动操作
```

 
这套组合的核心价值可以用一句话概括： **Superpowers 保证代码是怎么写出来的，gstack 保证写出来的东西能安全落地。** 两者之间没有争地盘的问题，而是形成了完整的闭环。

  
## 写在最后：AI 时代的“程序员”重新定义

 
回到开头那个问题：一个不爱写代码的程序员，还能叫程序员吗？

 
在 AI 编程助手出现之前，这个问题可能会让我焦虑。但现在我的答案是： **当然可以，而且可能更有价值。**

 
因为真正重要的从来不是“敲代码”这个动作本身，而是：

 
- 
你能不能把模糊的需求想清楚


- 
你能不能做出合理的技术决策


- 
你能不能判断方案的优劣


- 
你能不能把控项目的整体质量



 
这些能力，恰恰是我这些年积累下来的。而 AI 编程助手的出现，让我终于可以把这些能力发挥到极致，而不再被“写代码”这个执行环节卡住。

 
**这不是 AI 替代程序员，而是 AI 让每个程序员都能成为更好的自己。**

 
对我来说，这套 Claude Code 插件组合就是这个转变的最佳注脚。它让我从一个“不爱写代码的程序员”，变成了一个“用 AI 高效实现想法的创造者”。

 
如果你也有类似的困扰，或许可以试试这套工具。它可能不会让你爱上写代码，但一定会让你爱上实现想法的过程。
